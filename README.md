# ROMED8-2T BIOS analysis

> **Note: the contents of this repository were generated by an AI agent (Claude Code) under human direction.** The underlying analysis is real: disassembly, byte-level diffs, IFR decoding, and PE32 module sweeps were performed against actual ASRock BIOS images, and the conclusions are derived from that evidence. Because the prose, scripts, and finding documents were authored by a language model, individual claims should be independently verified before being relied upon, particularly for any action that affects hardware.

Reverse-engineering and reusable analysis tooling for ASRock Rack ROMED8-2T BIOS images (EPYC SP3 / AGESA Rome / Milan).

## Setup on a fresh clone

```bash
# 1. Clone:
git clone git@github.com:Banandana/romed8-2t-bios-analysis.git
cd romed8-2t-bios-analysis

# 2. Install tools (Arch / pacman example; adapt for your distro):
sudo pacman -S radare2 binwalk python python-pip
yay  -S uefitool ifrextractor-rs-bin                      # AUR
pip install --user --break-system-packages psptool

# 3. Build UEFIExtract from source (the AUR uefitool package omits it):
mkdir -p tools && cd tools
git clone --depth 1 https://github.com/LongSoft/UEFITool.git
cd UEFITool/UEFIExtract && cmake -B build -DCMAKE_BUILD_TYPE=Release . && cmake --build build -j
ln -sf "$PWD/build/uefiextract" ~/.local/bin/uefiextract
cd ../UEFIFind && cmake -B build -DCMAKE_BUILD_TYPE=Release . && cmake --build build -j
ln -sf "$PWD/build/uefifind" ~/.local/bin/uefifind
cd ../../..

# 4. (Optional) Register the radare2 MCP server for Claude Code agents:
r2pm -Uci r2mcp
# Then add to ~/.claude.json under mcpServers:
#   "radare2": {"command": "r2pm", "args": ["-r", "r2mcp"]}
```

## Re-downloading BIOS images

Public ROMED8-2T BIOSes are mirrored on archive.org. The ASRock site is JS-rendered and direct CDN URLs return 403, so use this pattern:

```bash
mkdir -p images && cd images
UA='Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0 Safari/537.36'
BASE='https://archive.org/download/asrock-server-ROMED8-2T'
for v in 3.70 3.80 3.90 4.10 L3.11; do
  curl -sS -L -A "$UA" -O "${BASE}/ROMED8-2T(${v})ROM.zip"
  unzip -o "ROMED8-2T(${v})ROM.zip"
done
cd ..
```

Each image extracts to `ROMD82T<version>` (33 MB). Note the bracketed parens in the URL — quote them in shells that interpret them.

## Onboarding a new BIOS image

```bash
# 1. Drop the .bin into images/ as ROMD82T<version> (e.g. ROMD82T3.80)
# 2. Extract (idempotent — re-runs are fast no-ops if already extracted):
mkdir -p extracted/all/P<version>
cp images/ROMD82T<version> extracted/all/P<version>/img.bin
(cd extracted/all/P<version> && uefiextract img.bin all)
# 3. Build per-version IFR reference (assuming you've also extracted IFR text per script):
python3 scripts/ifr_to_reference.py ifr/P<version>/_all docs/BIOS_REFERENCE_P<version>.md
# 4. Cross-version diff:
python3 scripts/diff_versions.py
```

A reusable `scripts/extract_bios.sh` wrapper is on the roadmap (see `docs/PLANNED_SUBAGENTS.md`).

## Re-running existing analyses

Every script in `scripts/` accepts a BIOS version (or two for diffs) as a parameter and re-runs cleanly. See header of each script for usage.

## Tools required

| Tool | Source | Purpose |
|------|--------|---------|
| `uefiextract` | LongSoft/UEFITool — built from source under `tools/` | Dump UEFI capsule into per-module FFS tree |
| `uefifind` | same | Search for GUIDs / patterns inside capsule |
| `ifrextractor` | AUR `ifrextractor-rs-bin` | Decode AMI Setup IFR |
| `radare2` | Arch `extra` package | Disassemble PE32 modules |
| `psptool` | `pip install --user --break-system-packages psptool` | Decode PSP firmware directory |
| `binwalk` | repo | Generic firmware structure inspection |

## What's NOT in this repo

- Raw BIOS images (large, possibly contain board-specific keys — keep local).
- The live rig's NVRAM dump (operationally sensitive — keep local).

## Versions analyzed

5 BIOS images on disk (gitignored): `L3.11`, `P3.70`, `P3.80`, `P3.90`, `P4.10`.

The rig in question runs `P3.70` on board rev 1.03.

## Key findings to date

See `BIOS_LATEST.md` for the synthesis. Short version: GPU-Gen3 cap on P3.70 is structural (HwInit-locked `LC_GEN4_EN_STRAP`); not user-correctable; P3.80 plausibly unlocks Gen4 for rev-1.03 boards (still being characterized).

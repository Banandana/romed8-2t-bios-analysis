# ROMED8-2T BIOS P3.90 — multi-GPU community-reports research

**Research date:** 2026-04-27
**Method:** WebFetch + WebSearch across the canonical ROMED8-2T community sources listed in the prompt.
**Bottom line:** **Nothing was found.** No public report — anywhere indexable — singles out P3.90 as the cause of a multi-GPU failure mode. Across every priority source surveyed, P3.90 simply does not appear in the discussion. The user's recollection cannot be corroborated from public traces.

This file documents the negative result with citations so that future searches do not have to repeat the work.

## Interpretation

Possibilities, in order of likelihood:

1. **The user's recollection conflated P3.80 with P3.90.** The only widely reported "multi-GPU has problems on this BIOS" datapoint is **slavaover's TID-24737 post about P3.80 on rev 1.02A**. Easy to mis-remember "3.80" as "3.90" — they're adjacent versions, both 2024-era. Strong prior: **the user is remembering the same datapoint already in CLAUDE.md, not a separate P3.90-specific report.**
2. **The user's recollection conflated P3.90 with the P4.10 + BMC 2.08 IPMI break.** That's the other prominent "this version is broken" community datapoint, but it's a BMC compat issue, not multi-GPU.
3. **A real but un-indexed P3.90 issue exists** in a closed Discord, a Slack channel, or a deleted/private post. Cannot rule out, but no public evidence.
4. **P3.90 is uneventful for multi-GPU.** Possible — if it was a routine AGESA bump with no observable user-visible regression for this workload, there'd be little reason for anyone to post.

Without flashing or rig-side empirical test (both off-limits per project constraints), there's no way to discriminate between (1)/(2)/(4). Recommend treating (1) as the working hypothesis until disproven.

## Severity assessment for an 8x RTX 3090 vLLM rig

**Found no specific evidence that P3.90 would harm an 8x GPU vLLM rig** beyond the general risks of any BIOS flash on a production system. Specifically:

- **No reported GPU-detection regression** in P3.90.
- **No reported bifurcation regression** in P3.90.
- **No reported BMC/IPMI compat break** at P3.90 (the IPMI break starts at P4.10).
- **No reported AGESA / link-training regression** in P3.90.
- The known **rev-1.03 Gen4 fix lands in P3.80**, so by P3.90 (released ~6 months later in Aug 2024), if the multi-GPU Gen4 path was going to break, it would have been called out alongside slavaover's 3.80 datapoint or the 4.10 IPMI-flash thread. It wasn't.

The rig (rev 1.03) is in the population that benefited from the 3.80 unlock, not the population that struggled at 3.80.

## Recommendation re: P3.90 if P3.80 doesn't unlock Gen4

The previous decision tree in CLAUDE.md was **P3.70 → P3.80** if the user revisits no-flash. The question now is whether P3.90 is a viable alternate target.

**Verdict: P3.90 should remain on the table, but as a secondary target behind P3.80.** Rationale:

1. **No known P3.90-specific multi-GPU regression** — the headline concern is unsubstantiated.
2. **P3.80 is the documented unlock point** for rev 1.03 (slavaover datapoint, ASRock TID 24737). Any later version inherits the unlock unless explicitly reverted, and there's no evidence of reversion.
3. **P3.90's static-analysis profile is consistent with P3.80** — per `docs/CROSS_VERSION_DIFF.md`, `AmdNbioPcieDxe` and the descriptor synthesizer evolved across P3.80→P3.90, then froze P3.90==P4.10. This means P3.90's runtime AGESA is closer to P4.10 than to P3.80. If anything, P3.90 should be **more robust** than P3.80 for multi-GPU since it includes whatever AGESA fixes ASRock rolled forward.
4. **Avoid P4.10** because of the BMC 2.08 IPMI-flash interlock (already documented). P3.90 sidesteps this — it's the last version before the IPMI break.
5. **Caveat:** Without a P3.90 changelog (ASRock doesn't publish one and wiki mirrors don't have it), we don't know the AGESA version bump or what else changed. The static-analysis evidence is the only signal we have.

Order of preference if a flash ever happens:
1. **P3.80** — known good for rev 1.03 + multi-GPU (slavaover); minimal divergence from P3.70.
2. **P3.90** — no known regressions; bigger AGESA delta than P3.80 but pre-IPMI-break.
3. **P4.10** — only with simultaneous BMC update or external-programmer flash.
4. **L4.11** — private hotfix, ticket-only, not generally available.

**This is not advocacy for flashing** — flashing remains hard-constrained off. This is a contingency-plan refinement.

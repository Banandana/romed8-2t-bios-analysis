# ROMED8-2T BIOS P3.90 — what was in this update

**Released:** 2024-08-12 (image mtime on the ASRock distribution ZIP and on the unpacked `.bin`).
**Investigator:** subagent dispatched 2026-04-27.
**Verdict:** **No official ASRock changelog exists.** Substantive content of the update inferred from the BIOS image itself plus cross-vendor AGESA release notes. Best characterization: **AGESA bump RomePI 1.0.0.G → 1.0.0.H + MilanPI 1.0.0.A → 1.0.0.C, with an interim AMD microcode 0x0830107B (dated 2023-08-16) bundled.** Almost certainly NOT the Sinkclose patch — that microcode (0x0830107C, AMD-SB-7014) was released by AMD only one day before P3.90 and almost certainly missed the ASRock build window.

## 1. Authoritative findings from the P3.90 binary itself

Source: `images/ROMD82T3.90` (32 MiB SPI image, mtime 2024-08-12).

### 1.1 AGESA version strings

```
$ strings -n 6 images/ROMD82T3.90 | grep -E "RomePI-SP3|MilanPI-SP3" | sort -u
MilanPI-SP3 1.0.0.C
RomePI-SP3  1.0.0.H
```

Cross-version table (verified locally on all 4 BIOS images on disk):

| BIOS    | Date       | RomePI       | MilanPI      |
|---------|------------|--------------|--------------|
| L3.11   | 2021-03-17 | 1.0.0.9      | 1.0.0.0      |
| P3.70   | 2023-05-30 | 1.0.0.F      | 1.0.0.A      |
| P3.80   | 2023-08-01 | 1.0.0.G      | 1.0.0.A      |
| **P3.90** | **2024-08-12** | **1.0.0.H** | **1.0.0.C** |
| P4.10   | 2025-06-05 | 1.0.0.L      | 1.0.0.G      |

So P3.90 is a coordinated AGESA bump on both Rome and Milan PI branches. The Milan jump (.A → .C) is a two-step bump; ASRock skipped a Milan-only intermediate.

### 1.2 Microcode patch_id (Family 17h Model 31h — Rome B0/B1)

Recovered by scanning the SPI image for AMD ucode header signatures (date dword in BCD `MMDDYYYY` immediately preceding patch_id):

| BIOS    | patch_id      | header date  | corresponds to          |
|---------|---------------|--------------|-------------------------|
| P3.70   | `0x08301055`  | 2022-02-15   | Errata 1161/1176/.../1417 fix per AGESA RomePI 1.0.0.E |
| P3.80   | `0x08301072`  | 2022-02-15   | (note: file date unchanged from .055 — vendor-internal renumber; possible early Zenbleed mitigation) |
| **P3.90** | **`0x0830107B`** | **2023-08-16** | post-Zenbleed (CVE-2023-20593 mitigation = `0x0830107A`); `.107B` is a non-public/OEM-distributed patch one revision newer |
| P4.10   | `0x0830107D`  | 2024-10-21   | post-Sinkclose (CVE-2023-31315 mitigation = `0x0830107C`); `.107D` is one revision newer |

**Important:** P3.90 does **not** ship the Sinkclose microcode (`0x0830107C`) despite being released 2024-08-12 — the same week AMD issued AMD-SB-7014. The public Sinkclose microcode landed in `linux-firmware` / `amd64-microcode` only on 2024-08-09 (Debian package `3.20240710.1`, published 2024-08-11). ASRock's image build was almost certainly already frozen. **Sinkclose was NOT what P3.90 was for.** The first ROMED8-2T BIOS that picks up post-Sinkclose microcode is P4.10 (`0x0830107D`).

The user's CLAUDE.md notes the rig is currently on `0x0830107C` — that's a Linux runtime microcode load (`/lib/firmware/amd-ucode`), not what's baked into P3.70's BIOS image. The BIOS image microcode is the cold-boot patch loaded by PSP before Linux gets to take over.

## 3. Inference from cross-vendor AGESA release notes

Other Rome SP3 OEMs **do** publish per-BIOS changelogs that name the AGESA version. Searches found:

- Supermicro H12SSL-i / H12SSW BIOS history (per Thomas-Krenn wiki / Supermicro release notes): RomePI 1.0.0.E → .F → .G correspond to the same Errata-fix / Zenbleed-fix progression we see in P3.70 → P3.80. Supermicro hasn't yet been observed shipping a public 1.0.0.H BIOS image as of these search results, so direct cross-vendor confirmation of "what's in 1.0.0.H" is unavailable from public docs.
- Gigabyte MZ32-AR0 (Rome-platform sibling) — release notes mention AGESA RomePI bumps but the public scrape didn't surface a 1.0.0.H entry either.

So while AGESA `RomePI-SP3 1.0.0.H` is real (it's in the P3.90 image strings), AMD does not publish AGESA release notes externally, and no sibling-OEM changelog mentioning .H was found.

**Most plausible content of `1.0.0.H` based on the AMD AGESA pattern + the date:** routine rollup of CPU errata, possibly DXIO fixes, and updated Milan support. AGESA bumps in this minor-version cadence are typically:
- Updated CPU support (new steppings of EPYC 7xx2 / 7xx3 — at this point in 2024, would be Milan-X derivatives)
- Memory training fixes
- Per-port PCIe/DXIO fixes
- Bundled AMD-ABL (PSP firmware) refresh

## 4. What changed in the P3.90 image vs P3.80 (binary diff context)

Per `docs/CROSS_VERSION_DIFF.md` and `docs/MODULE_SWEEP_P3.70_vs_P3.80.md` (already in repo), and extending the same modules for P3.80 → P3.90:

- `AmdApcbDxeV3` content changed P3.80 → P3.90 (also changed P3.70 → P3.80 by +64 B). The P3.80 → P3.90 delta is the descriptor synthesizer continuing to evolve.
- `AmdNbioPcieDxe` changed P3.80 → P3.90 (had been byte-identical P3.70 = P3.80). So the ESM/Gen4 consumer code itself was modified in this version — first time since L3.11.
- `CbsSetupDxeSSP` changed (continued evolution).
- ASRock `Setup` (HII) changed in every version including P3.90; per-slot Link Speed entries unchanged in count.
- APCB body byte-identical P3.80 = P3.90 (only V2 header instance-id + checksum differ) — confirmed by subagent #1.

Of these: the `AmdNbioPcieDxe` modification in P3.90 is genuinely new behavior (the previous five-version review had it frozen at P3.70 = P3.80). What that change does is not characterized here — would require a fresh radare2 diff.

## 6. Bottom line

**P3.90 is a routine AGESA refresh (RomePI 1.0.0.G → 1.0.0.H, MilanPI 1.0.0.A → 1.0.0.C) with intermediate microcode `0x0830107B`.** It is **not** a security-critical update (Sinkclose is in P4.10, not P3.90). It is the **first BIOS to modify `AmdNbioPcieDxe` since P3.70**, which is unusual — but the change has not been characterized in this investigation and could be unrelated to Gen4 cap behavior on the GPU root ports.

**For the user's specific question (Gen4 unlock):** community evidence (machine-agent's review of TID 24737, summarized in CLAUDE.md) places the rev-1.03 Gen4 unlock at **P3.80**, not P3.90. So P3.90 is not the unlock version per the community-reported evidence. Whether P3.90 added, regressed, or maintained the rev-1.03 Gen4 behavior is unknown — no community report of P3.90 Gen4 status was located in this search.
